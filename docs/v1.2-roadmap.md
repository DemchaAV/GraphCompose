# GraphCompose v1.2 Roadmap

This document tracks the v1.2 stabilization plan. The goal is not to add broad new features, but to make the existing Java-first document engine cleaner to use, easier to test, and safer to release.

The roadmap is split into 10 phases. Completed items reflect repository state; partial items remain release-polish work.

## Phase 0 - Release identity - Done

- [x] The next release ships as **v1.2.0** to match the project's current maturity and GitHub footprint.
- [x] `GraphCompose.document(...) -> DocumentSession -> DocumentDsl` is documented as the canonical authoring path.
- [x] Maven project coordinates use `io.github.demchaav:graphcompose:1.2.0`; JitPack uses `com.github.demchaav:GraphCompose:v1.2.0`.
- [x] Migration guide published at `docs/migration-v1-1-to-v1-2.md`.

## Phase 1 - Public API stabilization - In progress

Public packages:

- `com.demcha.compose`
- `com.demcha.compose.document.api`
- `com.demcha.compose.document.dsl`
- `com.demcha.compose.document.node`
- `com.demcha.compose.document.style`
- `com.demcha.compose.document.table`
- `com.demcha.compose.font`

Internal / advanced packages:

- `com.demcha.compose.engine.*`
- `com.demcha.compose.document.layout.*`
- `com.demcha.compose.document.backend.*`

Status:

- [x] README uses `GraphCompose.document(...)` as the public entry point.
- [x] `engine.*` is documented as internal / advanced, not the recommended authoring API.
- [x] `package-map.md` separates public and internal zones.
- [x] Runnable examples use the canonical public document API.
- [x] `DocumentSession` has closed-state lifecycle guards and idempotent `close()`.
- [x] Public DSL examples use `DocumentInsets`, `DocumentColor`, `DocumentStroke`, `DocumentTextStyle`, and `DocumentTable*`.
- [x] Remaining public `engine.*` allowlist leaks are being removed from `DocumentSession` and `font`.

## Phase 2 - Split `DocumentDsl` - Done

`DocumentDsl.java` is now a small facade. Focused builders live under `com.demcha.compose.document.dsl`:

```text
DocumentDsl.java
PageFlowBuilder.java
SectionBuilder.java
ModuleBuilder.java
ParagraphBuilder.java
ListBuilder.java
TableBuilder.java
ImageBuilder.java
ShapeBuilder.java
BarcodeBuilder.java
DividerBuilder.java
PageBreakBuilder.java
internal/
    BuilderSupport.java
    SemanticNameNormalizer.java
```

Current conversions are centralized in `document.layout.DocumentNodeAdapters`. The `document.dsl.internal` package currently contains builder and name helpers only.

## Phase 3 - Semantic layer purity - Done

- [x] Semantic nodes use renderer-neutral `DocumentLinkOptions`, `DocumentBookmarkOptions`, and `DocumentBarcodeOptions`.
- [x] PDF translation lives inside the fixed PDF backend/layout adapter layer.
- [x] Architecture guard tests keep `document.node.*` free of PDFBox imports.

## Phase 4 - Layout engine hardening - In progress

- [x] `BoxConstraints.UNBOUNDED_HEIGHT` centralizes the previous `1_000_000.0` sentinel.
- [x] `BoxConstraints.natural(width)` is the canonical factory for natural-height measurement.
- [x] `BoxConstraints.unboundedHeight(width)` remains as a compatibility alias.
- [x] Empty render throws a domain-specific `IllegalStateException`.
- [x] Pagination edge-case tests cover oversized atomic nodes, exact-fit content, leading/trailing page breaks, table row limits, nested sections, and module splits.

## Phase 5 - Rendering backend polishing - In progress

- [x] PDFBox code is isolated from `GraphCompose`, `document.api`, `document.dsl`, `document.node`, `document.layout`, and non-PDF backend contracts.
- [x] PDF metadata, protection, watermark, and header/footer options live behind `PdfFixedLayoutBackend.builder()`.
- [x] Debug guide-line overlays are available both through `PdfFixedLayoutBackend.builder()` and the session convenience path: `GraphCompose.document(...).guideLines(true)` / `DocumentSession.guideLines(true)`.
- [x] PDF render sessions own page-local content streams for the render pass.
- [x] Public font descriptors no longer materialize PDF/Word engine fonts directly.

## Phase 6 - Testing pyramid - In progress

- [x] Lifecycle tests cover closed-session and empty-document behavior.
- [x] Architecture guards cover public engine leaks, semantic PDFBox boundaries, and PDF backend isolation.
- [x] Layout snapshot and pagination tests cover the most important geometry cases.
- [x] Documentation/example tests render public README examples.
- [ ] Continue broadening public API compile coverage as new examples are added.

## Phase 7 - Performance and benchmarks - Done

- [x] Benchmark docs frame results as scenario-specific regression signals.
- [x] PR smoke and scheduled/manual full benchmark workflows exist.
- [x] JSON/CSV reports are produced as artifacts.
- [x] Median repeated-run mode is documented for local comparisons.
