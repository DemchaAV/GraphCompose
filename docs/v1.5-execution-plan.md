### A.6 — `expandWidth()` / `expandHeight()` shortcuts (DEFERRED → Phase B)

Effort: ~5–8 SP (re-estimated after first-week scoping). Moved to Phase B
because realising it cleanly needs a new `expandWidth/expandHeight` flag on
`ShapeNode` / `EllipseNode` / `ImageNode` records plus a measurement branch
in `LayoutCompiler` — i.e. the same kind of node-record extension already
planned for B.2 (`ShapeContainerNode`). Bundling them keeps the public
record API churn to a single release.

`addLine(thickness, color)` overload (originally bundled here) is
deferred with this work, since it needs an "expand to inner width" line by
default to be useful as a quick shortcut.

- [ ] (Phase B) Extend node records with `expandWidth` / `expandHeight` flags
- [ ] (Phase B) `*Builder.expandWidth()`, `expandHeight()` setters
- [ ] (Phase B) `LayoutCompiler` honours the flags during measurement
- [ ] (Phase B) `addLine(thickness, color)` overload backed by `expandWidth`
- [ ] (Phase B) Tests + recipe entry

### A.7 — `ListBuilder.addItem(label, Consumer<ListBuilder>)` (DEFERRED → Phase B)

Effort: ~6–8 SP (re-estimated). `ListNode` currently stores `List<String>` only —
there is no `children` hook for nested lists or rich-text items. Adding a
`ListItem` value type (with optional nested `ListNode`) is a public-record
change of the same shape as the other Phase B record extensions
(`ShapeContainerNode`, `expandWidth`-flag, `Transform`), so it batches with
those rather than fitting in the Phase A "sugar-only" bucket.

- [ ] (Phase B) Introduce `ListItem` value type carrying label + optional nested `ListNode`
- [ ] (Phase B) Update `ListNode` record (canonical signature change)
- [ ] (Phase B) `ListBuilder.addItem(label, Consumer<ListBuilder>)` API
- [ ] (Phase B) Layout pass: nested list inherits indent from parent
- [ ] (Phase B) Snapshot baseline `nested-list-2-levels.json`
- [ ] (Phase B) Recipe `docs/recipes/lists.md`

### B.3 — Layout: shape geometry pre-pass ✅

Effort: 5 SP — done in this Phase B beta. Architecture differs slightly
from the original plan: rather than a separate pre-pass that writes an
ECS-side `ShapeClipPath` component, the canonical compiler emits clip
geometry inline as fragment payloads. The end result is the same — clip
data is available before children render — but the data flows through the
existing `NodeDefinition.emitFragments(...)` pipeline and a new default
`emitOverlayFragments(...)` hook, keeping the canonical surface as the
single source of geometry.

- [x] `NodeDefinition` gains `default emitOverlayFragments(...)` returning
      empty list. Existing definitions inherit the default unchanged.
- [x] New payload pair on `BuiltInNodeDefinitions`:
      - [x] `ShapeClipBeginPayload(outline, policy, ownerPath)` — emitted
            from `ShapeContainerDefinition.emitFragments(...)` after the
            outline so it sits at fragment index 1
      - [x] `ShapeClipEndPayload(ownerPath)` — emitted from
            `ShapeContainerDefinition.emitOverlayFragments(...)` so it
            arrives after every layer fragment
- [x] `LayoutCompiler` invokes `definition.emitOverlayFragments(...)`
      after the children loop in both `compileStackedLayer(...)` (root
      level) and `compileNodeInFixedSlot(...)` STACK branch (nested).
      A new `compositeOverlayFragments(...)` helper mirrors
      `compositeDecorationFragments(...)` so the overlay path is
      multi-page-ready.
- [x] Architecture-guard: payloads live alongside other built-in payloads
      in `BuiltInNodeDefinitions` (canonical side); the existing
      `PublicApiNoEngineLeakTest` keeps engine.* imports out of the
      public surface. New `ShapeContainerInvariantsTest` adds
      cross-document begin/end pairing checks.
- **Files:** `NodeDefinition.java`, `LayoutCompiler.java`,
  `BuiltInNodeDefinitions.java` (extended ShapeContainerDefinition),
  `ShapeContainerInvariantsTest.java`

### B.5 — PDF render: clip path через graphics state ✅

Effort: 5 SP — done in this Phase B beta. Implementation differs slightly
from the original plan — instead of extending `PdfShapeFragmentRenderHandler`
with a `drawWithClippedChildren(...)` method, the clip is realised as
TWO new render handlers paired with the begin/end fragment markers from
B.3. This fits the existing flat-fragment dispatch model and keeps the
existing rect/ellipse handlers focused on their single responsibility.

- [x] New `PdfShapeClipBeginRenderHandler` — `saveGraphicsState() + add
      path + clip()`. Path is built from the `ShapeOutline` carried on
      the `ShapeClipBeginPayload`: ellipse → 4-curve Bezier ellipse;
      rounded rectangle → 4-curve corner Bezier; rectangle → `addRect()`.
      `CLIP_BOUNDS` always degrades to an axis-aligned rectangle clip
      regardless of outline kind.
- [x] New `PdfShapeClipEndRenderHandler` — `restoreGraphicsState()`.
- [x] Both handlers registered in
      `PdfFixedLayoutBackend.defaultHandlers()`.
- [x] Pair-balance covered by `ShapeContainerInvariantsTest`
      (`everyClipBeginInArbitraryDocumentHasMatchingEndOnSamePage`) +
      `ShapeContainerBuilderTest.everyClipBeginHasAMatchingClipEnd`.
- [x] PDF render smoke
      (`ShapeContainerBuilderTest.shapeContainerRendersToValidPdfBytesWithoutError`)
      ensures the new handlers dispatch cleanly through
      `session.toPdfBytes()`.
- [ ] Visual baselines: `circle-with-text`, `ellipse-with-overlay`,
      `rounded-rect-card` — deferred to Phase F polish.
- **Files:**
  `PdfShapeClipBeginRenderHandler.java`,
  `PdfShapeClipEndRenderHandler.java`,
  `PdfFixedLayoutBackend.java` (handlers list extended),
  `ShapeContainerBuilderTest.java` (extended).

### B.6 — DOCX backend: graceful fallback ✅

Effort: 2 SP — done. Decision differs slightly from the original plan: the
DOCX backend renders the container's *layers inline without the outline
frame* (Apache POI cannot express graphics-state path clipping, and
emulating an outline as a Word table border would be misleading because
it would only ever be a rectangle, not the actual ellipse / circle / 
rounded-rectangle the author asked for).

- [x] `DocxSemanticBackend.writeShapeContainer(...)` recursively walks
      `ShapeContainerNode.children()` via the existing `writeNode(...)`
      pipeline.
- [x] One-time SLF4J warning per export pass: `docx.export.shape-container-fallback`,
      reset at the start of each `export(...)` call so each session sees
      it at least once. (No dedicated `BackendCapabilityWarning` type —
      existing backends use SLF4J for the same purpose.)
- [x] Documented in `canonical-legacy-parity.md` under "Surfaces and
      structure": "Shape-as-container (clipped) — Partial — DOCX renders
      layers inline without the outline frame; PDF honours all three
      clip policies."
- [x] Test `DocxSemanticBackendTest.shapeContainerExportsLayersInlineWithoutOutline`
      verifies the fallback produces a valid DOCX whose paragraphs
      include the layer text.
- **Files:** `DocxSemanticBackend.java`, `DocxSemanticBackendTest.java`,
